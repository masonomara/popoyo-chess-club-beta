Read `docs/01-goal.md` thoroughly before doing anything else. Do not proceed until you have read and understood the full document.

## Your task

Generate two files — `docs/schema.sql` and `docs/seed.sql` — then add a `gen:types` script to `package.json`, then summarize what you created and give instructions for running it.

---

## Philosophy

This is the entire architecture step. Everything downstream — queries, server actions, components, UI — derives from the schema you write here. `app/types/database.ts` will be generated from this schema and is the single source of truth for all types in the codebase. No type is defined anywhere else. If code needs a cast, the schema is wrong — fix the schema, not the code.

Types are executable contracts, not documentation. Schema-first means: define the data model once, generate everything from it, and never let application code drift from what the database actually holds.

---

## Step 1 — Generate `docs/schema.sql`

Write a complete Postgres schema covering everything described in `docs/01-goal.md`. The schema must include:

**Tables**
- One table per entity — no exceptions for "simple" entities
- Every column has an explicit type and a `NOT NULL` constraint unless null is a meaningful value for that column
- Apply `UNIQUE` constraints wherever the data model requires uniqueness
- Use `UUID` primary keys with `DEFAULT gen_random_uuid()`
- Use `TIMESTAMPTZ` for all timestamps — include `created_at TIMESTAMPTZ NOT NULL DEFAULT now()` on every table

**Foreign keys**
- Declare every relationship with an explicit `FOREIGN KEY` constraint
- Use `ON DELETE CASCADE` unless there is a specific reason to use `SET NULL` or `RESTRICT` — document that reason in a comment

**Row Level Security**
- Run `ALTER TABLE [table] ENABLE ROW LEVEL SECURITY;` on every table
- Write explicit `CREATE POLICY` statements for SELECT, INSERT, UPDATE, and DELETE on every table
- Policies must reflect the roles and access rules described in `docs/01-goal.md` — do not write permissive catch-all policies; if a role cannot perform an operation, there should be no policy for it

**Postgres functions**
- Write `CREATE OR REPLACE FUNCTION` for any mutation that touches more than one table or that must run atomically
- These functions are called as RPCs from server actions — name them clearly

**Indexes**
- Add an index on every foreign key column
- Add indexes on columns that appear in frequent `WHERE` or `ORDER BY` clauses based on the flows described in `docs/01-goal.md`

Begin the file with:
```sql
-- Schema for [project name]. Generated by /schema.
-- Run in the Supabase SQL Editor before running seed.sql.
```

## Step 2 — Generate `docs/seed.sql`

Write realistic sample data that covers the full shape of the application:

- Include at least one row representing every user role described in `docs/01-goal.md`
- Include enough rows per table to make the app feel real when running locally — not 1 row, but enough to represent a real working state
- All seed data must respect every constraint in `schema.sql` — foreign keys, NOT NULL, UNIQUE, check constraints
- Use hardcoded UUIDs so foreign key relationships are explicit and auditable
- Use realistic values — not `test1`, `foo`, `bar`

Begin the file with:
```sql
-- Seed data for [project name]. Generated by /schema.
-- Run in the Supabase SQL Editor after schema.sql.
```

---

## Step 3 — Add `gen:types` to `package.json`

Add the following script to the `scripts` object in `package.json`:

```json
"gen:types": "supabase gen types typescript --project-id $SUPABASE_PROJECT_ID > app/types/database.ts"
```

`$SUPABASE_PROJECT_ID` is defined in `.env.local`. It is the project reference ID from the Supabase dashboard URL: `https://supabase.com/dashboard/project/[project-id]`.

---

## Step 4 — Summarize and give instructions

After generating all files, output:

1. A table listing every table in the schema: name, column count, one-line description of what it stores
2. Every Postgres function written, with a one-line description of what it does
3. Exact instructions for the user:
   - Open the [Supabase SQL Editor](https://supabase.com/dashboard)
   - Run `docs/schema.sql` first
   - Run `docs/seed.sql` second
   - Confirm `SUPABASE_PROJECT_ID` is set in `.env.local` (value is the project ref from the dashboard URL)
   - Run `npm run gen:types`
   - Confirm `app/types/database.ts` now contains the generated types
